# External Review: Konkani Transliterator

Persona: a senior engineer with a background on Google Translate's Indic transliteration systems, reviewing this project against the brief in `KONKANI-TRANSLITERATOR-BRIEF.md`.

---

Having spent years on Google's Indic transliteration stack, I'll say up front: this is a well-scoped little engine, and the greedy longest-match consonant-root parser is the *right* call at this scale. Full disclosure — production systems like ours eventually moved toward weighted FSTs and, later, statistical/neural transliteration models precisely because greedy rule matching falls apart once you're trying to cover dozens of dialects, code-mixed input, and millions of daily users typing garbage. But those systems solve a different problem: ranking *plausible* outputs over an ambiguous space, often with a language model backstop. What you have here is a *closed, deterministic* orthographic mapping for one dialect with a small, fixed phoneme inventory — that's exactly the regime where a hand-rolled greedy parser is not just adequate but *preferable*. It's transparent, debuggable by a linguist without ML background, and every rule traces to a line of code. I'd only reach for an FST if the rule table started needing weighted alternatives or backtracking — which brings me to the actual problems.

The consonant root extraction (`CONSONANT_ROOTS`, sorted by length descending) is a genuinely nice piece of engineering — deriving roots by stripping the inherent "a" from the same table used for display, rather than maintaining two parallel tables, is the sort of DRY choice that keeps a rule table honest as it grows. Same for reusing `VOWEL_SIGNS` bidirectionally. That data modeling will scale fine to more consonants, more vowel signs, even a second dialect variant, *as long as the alphabet stays one-token-in, one-token-out*. Where it will NOT scale gracefully is exactly the two problems flagged in the brief: the t/th retroflex-vs-dental collision and the gh-as-separator conjunct problem. Both are symptoms of the same structural gap — the parser has no notion of *lookahead-sensitive rule selection*. A real transliteration engine handles "context conditions the output" via either (a) an FST with epsilon transitions and priority-ordered arcs, or (b) simple lookahead rules bolted onto the greedy scan (e.g., "if root+h is followed by a vowel and h is not itself a listed root, treat h as a separator, not part of the conjunct"). The second is trivially implementable in this codebase's current shape — you don't need a rewrite, you need one more branch in the consonant-matching loop that peeks past a matched "gh"/"dh"/"bh" to see whether a vowel or halant naturally follows, and prefers the split reading. That's a day of work, not an architecture change.

The t/th question is a different animal — it's not a parsing bug, it's a genuine phonemic-convention conflict, and I'd resist calling it a defect. My advice, from having litigated identical "how do we type retroflex vs dental" fights in other Indic input methods: don't try to disambiguate this contextually, because there's no local context that reliably tells you if a given "t" is meant as dental or retroflex in casual Konkani spelling — it's genuinely bimodal by user intent, not by phonetic environment. Ship the "casual mode vs strict Sudhaarit mode" toggle the brief already proposes, as a hard runtime switch on the rule table (literally two versions of the CONSONANT_ROOTS array, chosen at init), rather than trying to make one clever rule guess right. Anything cleverer will regress on some word for some user and you'll be firefighting anecdotes forever.

The anusvara collapse (ಂ → "m" always on the reverse path) is correctly identified as lossy-by-design rather than a bug, and I agree with that framing — you cannot round-trip a phoneme merger without external information, full stop. This is the same class of problem as sandhi/schwa-deletion loss in Devanagari romanization: accept the lossiness, don't chase it. If it ever matters, the fix isn't smarter code — it's a corpus-driven frequency prior on which surface form is more likely per word, which is genuinely overkill for a dialect tool at this scale.

The halant fix (stripping the trailing inherent "a" before the halant "continue," `transliteration.js` ~line 124-128) is correct and matches how a matra substitution is already handled a few lines below it — good, consistent pattern, and it closes a real round-trip bug (ದೇವ್ → "deev" not "deeva").

Code quality-wise, my only real nitpick: `kannadaToLatinTransliterate` rebuilds and re-sorts `sortedKeys` from the Map on every function call (line 119) — harmless at this input size, wasteful pattern, hoist it to module scope like `SORTED_VOWEL_KEYS` already is. Also the string-mutation approach (`result.endsWith("a"); result.slice(0,-1)`) that both the halant and matra branches now share is doing the "undo the inherent vowel" trick twice in slightly different call sites — worth factoring into one `stripInherentA(result)` helper before a third call site (conjuncts, maybe) needs the same logic and someone forgets to update all three.

App.jsx is unremarkable in the best sense — thin wrapper, no transliteration logic leaking into the UI, accessible tab/modal patterns. The TEST_WORDS/feedback-tab omission is fine to leave dark for now, but I'd flag for the roadmap: if the plan really is "collect feedback from native speakers," you want that loop writing structured corrections somewhere durable (even a dumb Google Form or Sheet) before you invest more engineering time guessing at the h-separator question, because that's fundamentally a data problem, not an algorithm problem — no amount of clever FST design substitutes for asking twenty native speakers how they actually type "bhurgo."

**Verdict:** solid, honest prototype for its actual scope. The architecture is right-sized — I would not recommend FSTs or statistical modeling for a single-dialect community tool with ~30 consonants and a few dozen known edge cases; that's solving a problem you don't have yet. The two flagged parsing gaps (gh-conjunct, t/th) are shallow fixes within the current design, not signs the greedy approach is running out of road. The one thing I'd insist on before this scales past a handful of users is turning "collect feedback" from an aspiration into an actual mechanism, because the remaining open questions (kh/ph/jh-as-separator, casual vs strict mode) are empirical questions about how Mangalorean Catholics actually type, not things you can resolve by staring at the parser harder.
